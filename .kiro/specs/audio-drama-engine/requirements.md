# Requirements Document

## Introduction

The Audio Drama Engine is an AI-powered application that transforms written stories into fully dramatized audio experiences. The system analyzes input stories, extracts characters and scenes, adapts the text into dialogue-heavy cinematic scripts on a scene-by-scene basis with context passing, and generates multi-layered audio using ElevenLabs TTS, Sound Effects, and Music APIs. Final audio is mixed with FFmpeg. Users can select from 10 drama styles (Anime, Noir, Cyberpunk, Horror, Dark Thriller, Fantasy Epic, Romance, Comedy, Documentary, Cinematic) that influence adaptation tone, voice casting, soundscape design, and musical scoring via rich style presets injected into LLM prompts. Output is organized as episodes with a built-in premium player UI reminiscent of streaming platforms. The system supports .txt, .md, .pdf, and .epub file uploads up to 5MB.

## Glossary

- **Story_Ingestion_Module**: The component responsible for accepting user-uploaded stories (in .txt, .md, .pdf, or .epub format up to 5MB) or generating original stories from user prompts.
- **Drama_Style**: A selectable aesthetic preset (Anime, Noir, Dark Thriller, Horror, Cyberpunk, Fantasy Epic, Romance, Comedy, Documentary, Cinematic) that influences script adaptation tone, voice characteristics, sound effect selection, and music scoring.
- **Style_Preset**: A structured configuration associated with each Drama_Style containing fields for narration_style, dialogue_style, music_preferences, ambient_preferences, sfx_style, pacing, and voice_aesthetic, injected into LLM prompts during combined scene adaptation.
- **Story_Analyzer**: The LLM-powered component that performs metadata extraction (Round 1), scene decomposition (Round 2), and coherence verification (Round 4) of the input story.
- **Scene_Adapter**: The LLM-powered component that performs combined scene adaptation (Round 3), processing one scene at a time with context passing to produce an AnnotatedScene containing dialogue, narration, action cues, SFX cues, and music cues in a single LLM call per scene.
- **Character_Voice_Mapper**: The component that assigns distinct ElevenLabs voice profiles to each character based on their described or inferred physical and personality traits.
- **Audio_Generator**: The component that calls ElevenLabs APIs to produce TTS audio, sound effects, and music tracks from the finalized script.
- **Audio_Mixer**: The FFmpeg-based component that combines TTS dialogue, sound effects, and music tracks into a final mixed audio output with proper volume balancing and timing.
- **Episode**: A discrete unit of the audio drama containing one or more scenes, used to break longer stories into consumable segments.
- **Intermediate_Format**: The structured data representation of the dramatized script including all dialogue lines, SFX cues, music cues, timing information, and metadata, used as input for audio generation.
- **Drama_Player**: The built-in UI component that presents episodes in a premium streaming-platform style and provides playback controls.
- **Narration**: Spoken text used only to convey information that cannot be communicated through dialogue, sound effects, music, or actions.
- **SFX_Duration_Type**: Classification of a sound effect as either momentary (a brief one-time sound) or ambient (a continuous sound persisting for a duration within a scene).
- **Creative_Mode**: A user-togglable option that, when enabled, allows the Scene_Adapter to make more liberal creative additions and alterations to enhance the cinematic quality of the audio drama, while preserving story fidelity.
- **Thumbnail_Generator**: The component that programmatically generates drama cover thumbnails using HTML Canvas or a Node.js canvas library, producing a 640x360px image with a gradient background, drama title, and style badge using style-specific color palettes.
- **Recap_Narration**: A brief "Previously on..." narration segment (1-2 sentences) generated by the LLM during episode organization and prepended to the start of episodes 2 and beyond in multi-episode dramas.

## Requirements

### Requirement 1: Story Input

**User Story:** As a user, I want to provide stories either by uploading files in multiple formats or by entering prompts that generate original stories, so that I can create audio dramas from any source material.

#### Acceptance Criteria

1. WHEN a user uploads a file containing a story in .txt, .md, .pdf, or .epub format, THE Story_Ingestion_Module SHALL accept the file and extract the story text for processing.
2. WHEN a user submits a prompt describing a story concept along with optional style and length preferences, THE Story_Ingestion_Module SHALL generate an original story using an LLM and present it to the user for confirmation before proceeding.
3. IF the uploaded file is empty or contains no extractable text, THEN THE Story_Ingestion_Module SHALL display a descriptive error message indicating the file is invalid.
4. IF the uploaded file exceeds the maximum supported size of 5MB, THEN THE Story_Ingestion_Module SHALL display an error message stating the size limit.
5. IF the uploaded file format is not .txt, .md, .pdf, or .epub, THEN THE Story_Ingestion_Module SHALL display an error message listing the four supported formats.
6. WHEN a user submits a prompt for story generation with a selected Drama_Style, THE Story_Ingestion_Module SHALL pass the style parameter to the LLM so the generated story matches the selected drama style.

### Requirement 2: Drama Style Selection

**User Story:** As a user, I want to select from a rich palette of drama styles for my audio drama, so that the tone, voice casting, sound design, and music match my preferred aesthetic.

#### Acceptance Criteria

1. THE Drama_Player SHALL present the user with a list of available Drama_Style options including Anime, Noir, Dark Thriller, Horror, Cyberpunk, Fantasy Epic, Romance, Comedy, Documentary, and Cinematic.
2. WHEN a user selects a Drama_Style, THE Story_Analyzer SHALL apply that style's tonal characteristics throughout all subsequent analysis and adaptation rounds.
3. WHEN no Drama_Style is selected by the user, THE Story_Analyzer SHALL infer an appropriate default style from the full palette of 10 styles based on the story's genre, themes, and narrative tone.
4. WHEN a Drama_Style is applied, THE Scene_Adapter SHALL load the corresponding Style_Preset and inject its narration_style, dialogue_style, music_preferences, ambient_preferences, sfx_style, pacing, and voice_aesthetic fields into the combined scene adaptation prompt.
5. THE system SHALL define a Style_Preset configuration for each of the 10 Drama_Style options containing fields for narration_style, dialogue_style, music_preferences, ambient_preferences, sfx_style, pacing, and voice_aesthetic.

### Requirement 3: Multi-Round Story Analysis with Scene-by-Scene Adaptation

**User Story:** As a user, I want the system to thoroughly analyze my story and adapt it scene by scene with context passing, so that no narrative detail is lost and each scene benefits from awareness of what came before.

#### Acceptance Criteria

1. WHEN a story is submitted for processing, THE Story_Analyzer SHALL perform a metadata extraction round (Round 1) as a single LLM call that identifies characters (with physical descriptions, personality traits, and relationships), settings, time periods, themes, and narrative arc.
2. WHEN metadata extraction is complete, THE Story_Analyzer SHALL perform a scene decomposition round (Round 2) as a single LLM call that breaks the story into discrete scenes with identified locations, participating characters, and scene-level mood.
3. WHEN scene decomposition is complete, THE Scene_Adapter SHALL perform a combined scene adaptation round (Round 3) that iterates scene by scene, where each LLM call receives the full metadata as persistent context, the previous scene's AnnotatedScene output (or "none" for the first scene), and the raw text for only the current scene, producing one AnnotatedScene per call containing dialogue, narration, action cues, SFX cues, and music cues.
4. WHEN all scenes are adapted, THE Scene_Adapter SHALL concatenate the individual AnnotatedScene outputs into the full AnnotatedScript.
5. WHEN combined scene adaptation is complete, THE Story_Analyzer SHALL perform a coherence verification round (Round 4) as a single LLM call that reviews the full adapted script against the original story to detect information loss, inconsistencies, pacing issues, and creative appropriateness.

### Requirement 4: Combined Scene Adaptation

**User Story:** As a user, I want the system to create a cinematic dramatized script with sound effects and music cues in a single adaptation pass per scene, so that all audio elements are coherently designed together.

#### Acceptance Criteria

1. THE Scene_Adapter SHALL produce AnnotatedScene output where dialogue is the primary vehicle for conveying story information, with SFX cues and music cues integrated in the same LLM call.
2. THE Scene_Adapter SHALL use Narration only to convey information that cannot be communicated through dialogue, sound effects, music, or character actions.
3. WHEN adapting dialogue, THE Scene_Adapter SHALL include expression tags and emotional delivery cues for each line to guide TTS generation via the ElevenLabs API.
4. THE Scene_Adapter SHALL preserve all plot points, character motivations, and thematic elements from the original story in the adapted script.
5. WHEN a scene contains character actions, THE Scene_Adapter SHALL represent those actions through a combination of dialogue, sound effects, and brief action descriptions rather than lengthy narration.
6. WHEN adapting a scene, THE Scene_Adapter SHALL generate SFX cues for explicit sound mentions, inferred environmental and action sounds, and emotional ambience sounds, classifying each as momentary or ambient with appropriate timing and duration.
7. WHEN adapting a scene, THE Scene_Adapter SHALL generate music cues selectively at moments of emotional weight, dramatic tension, or narrative turning points, using silence as a deliberate dramatic tool.
8. WHEN adapting a scene, THE Scene_Adapter SHALL specify transition types for music cues including fade-in, fade-out, crossfade, and hard cut.
9. WHEN music overlaps with dialogue in a scene, THE Scene_Adapter SHALL mark the music cue as underscore with reduced volume priority relative to dialogue.

### Requirement 5: Character Voice Assignment

**User Story:** As a user, I want each character to have a distinct voice that matches their described or inferred qualities, so that the audio drama feels like a multicast production.

#### Acceptance Criteria

1. WHEN character metadata is extracted, THE Character_Voice_Mapper SHALL assign a distinct ElevenLabs voice profile to each identified character based on their described or inferred age, gender, personality, and physical traits.
2. THE Character_Voice_Mapper SHALL assign a dedicated narrator voice that is distinct from all character voices.
3. THE Character_Voice_Mapper SHALL maintain consistent voice assignments for each character across all scenes and episodes of the audio drama.
4. WHEN the selected Drama_Style changes voice aesthetic expectations, THE Character_Voice_Mapper SHALL adjust voice profile selections to match the style (e.g., grittier voices for Noir, exaggerated expressions for Anime, warm tones for Romance, authoritative delivery for Documentary).

### Requirement 6: Sound Effect Cue Generation

**User Story:** As a user, I want the system to add realistic sound effects based on both explicit mentions and contextual inference, so that the audio drama has an immersive soundscape.

#### Acceptance Criteria

1. WHEN a scene explicitly mentions a sound (e.g., a door slamming, glass breaking), THE Scene_Adapter SHALL generate a corresponding SFX cue in the AnnotatedScene.
2. WHEN a scene describes actions or environments that imply sounds not explicitly mentioned (e.g., footsteps when a character walks, rain when a storm is described), THE Scene_Adapter SHALL infer and generate appropriate SFX cues.
3. FOR each SFX cue, THE Scene_Adapter SHALL classify the SFX_Duration_Type as either momentary or ambient.
4. WHEN an SFX cue is classified as ambient, THE Scene_Adapter SHALL specify the duration for which the sound persists within the scene.
5. WHEN an SFX cue is classified as momentary, THE Scene_Adapter SHALL specify the trigger point relative to the dialogue or action timeline.
6. THE Scene_Adapter SHALL infer emotional ambience sounds (e.g., a ticking clock during a tense moment, wind howling during isolation) based on the scene's mood and context.

### Requirement 7: Cinematic Music Scoring

**User Story:** As a user, I want the system to add cinematic music at appropriate moments, so that the audio drama has the emotional impact of a film score.

#### Acceptance Criteria

1. WHEN a scene has significant emotional weight, dramatic tension, or a narrative turning point, THE Scene_Adapter SHALL generate a music cue specifying mood, intensity, and approximate duration.
2. THE Scene_Adapter SHALL place music cues selectively, using silence and absence of music as deliberate dramatic tools rather than scoring every scene.
3. WHEN a Drama_Style is applied, THE Scene_Adapter SHALL select music moods and instrumentation consistent with that style's music_preferences from the Style_Preset (e.g., orchestral for Fantasy Epic, soft piano for Romance, synthwave for Cyberpunk, jazz for Noir).
4. THE Scene_Adapter SHALL specify transition types for music cues including fade-in, fade-out, crossfade, and hard cut.
5. WHEN music overlaps with dialogue, THE Scene_Adapter SHALL mark the music cue as underscore with reduced volume priority relative to dialogue.

### Requirement 8: Audio Generation via ElevenLabs APIs

**User Story:** As a user, I want the system to generate high-quality audio for voices, sound effects, and music using ElevenLabs APIs, so that the output sounds professional and polished.

#### Acceptance Criteria

1. WHEN the Intermediate_Format is finalized, THE Audio_Generator SHALL call the ElevenLabs TTS API to generate speech audio for each dialogue line and narration segment using the assigned voice profiles and expression tags.
2. WHEN SFX cues are present in the Intermediate_Format, THE Audio_Generator SHALL call the ElevenLabs Sound Effects API to generate corresponding audio clips.
3. WHEN music cues are present in the Intermediate_Format, THE Audio_Generator SHALL call the ElevenLabs Music API to generate corresponding music tracks.
4. IF an ElevenLabs API call fails, THEN THE Audio_Generator SHALL retry the call up to 3 times with exponential backoff before reporting the failure to the user.
5. THE Audio_Generator SHALL cache generated audio assets to avoid redundant API calls for identical requests within the same story processing session.

### Requirement 9: Audio Mixing

**User Story:** As a user, I want all audio layers to be mixed into a polished final output, so that dialogue, sound effects, and music blend together seamlessly.

#### Acceptance Criteria

1. WHEN all audio assets are generated, THE Audio_Mixer SHALL combine dialogue, SFX, and music tracks into a single mixed audio file per scene using FFmpeg.
2. THE Audio_Mixer SHALL prioritize dialogue volume over SFX and music, ensuring speech remains clearly audible at all times.
3. WHEN music cues are marked as underscore, THE Audio_Mixer SHALL reduce music volume relative to dialogue by the amount specified in the music cue.
4. THE Audio_Mixer SHALL apply the transition types specified in music cues (fade-in, fade-out, crossfade, hard cut) at the designated timing points.
5. WHEN ambient SFX are present, THE Audio_Mixer SHALL loop or extend the ambient audio to cover the specified duration within the scene.
6. THE Audio_Mixer SHALL concatenate mixed scene audio files into complete Episode audio files in the correct scene order.

### Requirement 10: Episodic Organization

**User Story:** As a user, I want longer stories to be broken into episodes with recap narration, so that I can consume the audio drama in manageable segments with continuity.

#### Acceptance Criteria

1. WHEN a story contains more scenes than fit a single episode, THE Story_Analyzer SHALL group scenes into episodes based on narrative arc boundaries and pacing.
2. THE Story_Analyzer SHALL assign each Episode a title and a brief synopsis derived from the scenes it contains.
3. WHEN a story is short enough to fit a single episode, THE Story_Analyzer SHALL produce a single Episode containing all scenes.
4. THE Story_Analyzer SHALL maintain narrative continuity across episode boundaries, avoiding mid-scene splits.
5. WHEN organizing a multi-episode drama, THE Story_Analyzer SHALL generate a Recap_Narration of 1-2 sentences summarizing the previous episode's events for each episode after the first.
6. WHEN an episode has a Recap_Narration, THE Audio_Generator SHALL prepend a "Previously on..." narration segment at the start of that episode's audio.

### Requirement 11: Drama Player UI

**User Story:** As a user, I want a premium streaming-platform-style player to browse and play my audio dramas, so that the experience feels like a high-quality entertainment application.

#### Acceptance Criteria

1. THE Drama_Player SHALL display a library view showing all generated audio dramas with thumbnails, title, style badge, and episode count.
2. WHEN a user selects an audio drama, THE Drama_Player SHALL display an episode list with episode titles, synopses, and durations.
3. THE Drama_Player SHALL provide playback controls including play, pause, skip forward 15 seconds, skip backward 15 seconds, and a seekable progress bar.
4. WHEN an episode finishes playing, THE Drama_Player SHALL automatically advance to the next episode in the series.
5. THE Drama_Player SHALL display the currently playing episode title, drama title, and elapsed time during playback.
6. THE Drama_Player SHALL persist playback position using browser localStorage with the key format `drama_${dramaId}_episode_${episodeId}` so that a user returning to a previously started episode resumes from where playback was last active.

### Requirement 12: Intermediate Format Structure

**User Story:** As a developer, I want a well-defined intermediate format between the script adaptation and audio generation phases, so that the pipeline stages are decoupled and debuggable.

#### Acceptance Criteria

1. THE Scene_Adapter SHALL output the Intermediate_Format as a structured data object containing episode metadata, scene definitions, dialogue lines with voice assignments and expression tags, SFX cues with timing and duration type, and music cues with mood and transition specifications.
2. THE Intermediate_Format SHALL include a unique identifier for each dialogue line, SFX cue, and music cue to enable traceability between script and generated audio assets.
3. THE Audio_Generator SHALL consume the Intermediate_Format as its sole input, with no dependency on the original story text or intermediate analysis artifacts.

### Requirement 13: Pipeline Progress and Error Reporting

**User Story:** As a user, I want to see granular progress of my audio drama generation and be informed of any errors, so that I know the status of my request at all times.

#### Acceptance Criteria

1. WHILE the pipeline is processing a story, THE Drama_Player SHALL display the current processing stage (e.g., "Analyzing story", "Adapting scene 3 of 8", "Producing audio", "Mixing audio") and an estimated progress indicator.
2. IF a processing stage fails after exhausting retries, THEN THE Drama_Player SHALL display a descriptive error message identifying the failed stage and the nature of the failure.
3. WHEN processing completes successfully, THE Drama_Player SHALL notify the user and make the generated audio drama immediately available for playback.
4. WHILE the audio generation stage is active, THE Pipeline_Orchestrator SHALL emit granular sub-progress updates (e.g., "Generating voice 3 of 12", "Generating sound effect 5 of 8") to the Drama_Player.

### Requirement 14: Creative Mode

**User Story:** As a user, I want to enable a Creative Mode that allows the system to make more expressive creative choices in sound effects, music, and script adaptation, so that the audio drama feels more atmospheric and cinematic beyond what the source text explicitly describes.

#### Acceptance Criteria

1. THE Drama_Player SHALL present Creative_Mode as a togglable option that the user can enable or disable before or during story processing.
2. WHILE Creative_Mode is enabled, THE Scene_Adapter SHALL add atmospheric and mood-driven sound effects that enhance the emotional tone of a scene even when those sounds are not directly described or implied by the text (e.g., unsettling crow sounds in a horror scene, distant screeching to convey uneasiness).
3. WHILE Creative_Mode is enabled, THE Scene_Adapter SHALL add mood-enhancing music cues more liberally to amplify the emotional atmosphere of scenes, including scenes that would not otherwise receive music cues.
4. WHILE Creative_Mode is enabled, THE Scene_Adapter SHALL alter dialogue and narration more freely to enhance cinematic quality, including rephrasing lines for dramatic effect and adding expressive beats, while preserving the original meaning and information content.
5. WHILE Creative_Mode is enabled, THE Scene_Adapter SHALL ensure that all creatively added sound effects remain contextually appropriate to the scene's setting, genre, and mood, and are not incongruent with the narrative context.
6. WHILE Creative_Mode is enabled, THE Scene_Adapter SHALL ensure that all creatively added music cues are stylistically consistent with the selected Drama_Style and the emotional tone of the scene.
7. WHILE Creative_Mode is enabled, THE Scene_Adapter SHALL NOT introduce any additions or alterations that cause loss of information, inconsistency, or contradiction with the original story.
8. WHILE Creative_Mode is enabled, THE Story_Analyzer SHALL include a dedicated creative fidelity check during the coherence verification round (Round 4) that validates all creative additions and alterations against the original story for information completeness, narrative consistency, and absence of contradictions.
9. WHILE Creative_Mode is disabled, THE Scene_Adapter SHALL operate under standard inference and adaptation rules without expanded creative liberties.

### Requirement 15: Thumbnail Generation

**User Story:** As a user, I want each audio drama to have a visually distinctive thumbnail, so that the library view looks polished and each drama is easily identifiable.

#### Acceptance Criteria

1. WHEN a new drama is created, THE Thumbnail_Generator SHALL programmatically generate a 640x360px (16:9) thumbnail image using HTML Canvas or a Node.js canvas library.
2. THE Thumbnail_Generator SHALL render each thumbnail with a gradient background, the drama title text, and a style badge.
3. THE Thumbnail_Generator SHALL use style-specific color palettes for the gradient background: Noir (charcoal to amber), Anime (pink to lavender), Horror (black to blood red), Cyberpunk (purple to neon cyan), Dark Thriller (navy to steel gray), Fantasy Epic (dark green to gold), Romance (blush pink to coral), Comedy (bright yellow to orange), Documentary (cool gray to white), Cinematic (deep indigo to silver).
4. THE Thumbnail_Generator SHALL generate the thumbnail once at drama creation time and store it for use in the library view.

### Requirement 16: LLM Prompt Strategy

**User Story:** As a developer, I want well-defined prompt templates for each LLM round, so that the system produces consistent, high-quality structured output.

#### Acceptance Criteria

1. THE Story_Analyzer SHALL use an Analysis Prompt template for Round 1 that includes a system role definition, the full story text, the selected style, a JSON output schema, and an instruction to output only valid JSON.
2. THE Story_Analyzer SHALL use a Scene Decomposition Prompt template for Round 2 that includes a system role definition, the story text combined with extracted metadata, and scene boundary rules.
3. THE Scene_Adapter SHALL use a Combined Scene Adaptation Prompt template for Round 3 that includes a system role definition, the full metadata, the Style_Preset for the selected style, the previous scene's AnnotatedScene output (or "none"), the current scene's raw text, creative mode instructions, detailed rules for dialogue, SFX, and music generation, and the AnnotatedScene JSON schema.
4. THE Story_Analyzer SHALL use a Coherence Verification Prompt template for Round 4 that includes a system role definition, the original story text, the extracted metadata, the full adapted script, and checks for missing characters, missing plot points, continuity errors, and creative appropriateness.
